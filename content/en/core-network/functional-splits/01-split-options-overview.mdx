## Why this matters

Traditional base stations bundle everything: RF, baseband, control, and backhaul. Splitting enables flexibility — run parts centrally, parts at the edge, mix vendors.

But where to split? Each option trades off fronthaul bandwidth, latency tolerance, and pooling gain. The industry converged on Option 7-2x for O-RAN, but understanding why requires knowing the alternatives.

</section>

### Split options

```
3GPP Split Options (TR 38.801):

Split    │ Boundary                    │ What's Centralized
─────────┼─────────────────────────────┼─────────────────────
Option 1 │ Above RRC                   │ Nothing (all at site)
Option 2 │ RRC / PDCP                  │ RRC
Option 3 │ PDCP / RLC                  │ RRC, PDCP
Option 4 │ RLC-high / RLC-low          │ RRC, PDCP, RLC-high
Option 5 │ RLC / MAC                   │ RRC, PDCP, RLC

│                                                          │
│  Option 8 ◄─────────── Fronthaul BW ───────────► Option 1
│  (most BW)                                     (least BW)

│  Option 8 ◄─────────── Latency strict ─────────► Option 1
│  (strictest)                                   (relaxed)
│                                                          │
│  Option 8 ◄─────────── Pooling gain ───────────► Option 1
│  (most gain)                                   (no gain)

### Why Option 7-2x

```
Option 7-2x Rationale:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  Requirements:                                          │
│  1. Multi-vendor interoperability (need clean boundary) │
│  2. Centralized scheduling (pooling, CoMP)             │
│  3. Manageable fronthaul bandwidth                      │
│  4. RU simplicity (cost, power)                        │

│  Option 7-2x achieves:                                  │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  ✓ Clean interface: IQ samples + control        │    │
│  │  ✓ Scheduler at DU: coordinated, pooled         │    │
│  │  ✓ ~25 Gbps fronthaul: 25G Ethernet works       │    │
│  │  ✓ Simple RU: IFFT + RF only                    │    │
│  │                                                  │    │

## Apply

**Split option selection:**

| Use Case | Recommended Split | Reason |
|----------|-------------------|--------|
| Macro cell | Option 2 + 7-2x | CU pooling + O-RAN |
| Small cell | Option 2 + 7-2x | Same ecosystem |
| Massive MIMO | 7-2x Cat-B | Fronthaul efficiency |
| Rural (limited transport) | Option 2 only | Minimize FH BW |
| Cloud RAN | 7-2x | Maximum pooling |

**Fronthaul sizing:**

```bash
# Calculate fronthaul for 100 MHz TDD, 64T64R, Cat-B
# IQ bits: 2 (I+Q) × 9 bits × 12 subcarriers × 273 PRBs × 14 symbols
# Per ms: ~7.5 Gbps per direction (before compression)
# With BFP: ~4 Gbps per direction
# Bidirectional: ~8-10 Gbps typical, 25 Gbps peak

# Recommended transport:
# - 25G Ethernet for single sector
# - 100G Ethernet for aggregated sites
```

**Interface summary:**

| Interface | Protocol | Location |
|-----------|----------|----------|
| F1 | GTP-U/SCTP | CU ↔ DU |
| Open Fronthaul | eCPRI | DU ↔ RU |
| E1 | SCTP | CU-CP ↔ CU-UP |
| E2 | SCTP | Near-RT RIC ↔ CU/DU |

</section>

<section class="lesson-pitfalls">

## Pitfalls

1. **Underestimating fronthaul.** 64T64R needs 25G+ even with compression. Plan transport capacity.

2. **Latency budget blown.** Option 7-2x needs < 250 µs RTT. Fiber length and switch hops add up.

3. **Mixing splits.** Different vendors may implement slightly different 7-2x. Verify interoperability.

4. **Ignoring timing.** All splits need precise sync. Option 7-2x needs ±1.5 µs phase for TDD.

5. **Cat-A/B mismatch.** DU must match RU category. Wrong pairing breaks beamforming.

6. **Compression artifacts.** Aggressive IQ compression degrades EVM. Balance bandwidth vs quality.

7. **CPRI lock-in.** Option 8 (CPRI) is proprietary per vendor. O-RAN 7-2x is the open choice.

8. **Upgrade complexity.** Changing split requires new hardware at both ends. Plan for lifecycle.
