## Why this matters

Open Fronthaul promises vendor flexibility. But the transport network must meet stringent requirements — bandwidth, latency, jitter, and synchronization — or the system fails.

Understanding transport requirements is essential for fronthaul network design. Get it wrong, and the RAN doesn't work.

</section>

## How it works

### Bandwidth calculation

```
Fronthaul Bandwidth Requirements:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  Uncompressed IQ (16-bit I + 16-bit Q):                 │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  BW = PRBs × 12 subcarriers × 2 (I+Q)           │    │
│  │       × 32 bits × 14 symbols × SCS/15           │    │
│  │                                                  │    │
│  │  100 MHz (273 PRBs, SCS 30 kHz):                │    │
│  │  273 × 12 × 2 × 32 × 14 × 2 = ~3.7 Gbps         │    │
│  │                                                  │    │
│  │  Per 4T4R antenna: 3.7 × 4 = ~14.7 Gbps         │    │
│  │  Per carrier: × MIMO layers                     │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  Compressed (BFP 9-bit):                                │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  Compression ratio: ~9/16 = 56%                 │    │
│  │  BFP overhead: +4 bits per PRB                  │    │
│  │                                                  │    │
│  │  100 MHz, 4T4R: ~8.5 Gbps                       │    │
│  │  + eCPRI overhead (~5%): ~9 Gbps                │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  Typical Interface Sizing:                              │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  Single-sector 4T4R:  10 GbE (with compression) │    │
│  │  3-sector site:       25 GbE or 2×10 GbE        │    │
│  │  64T64R mMIMO:        25 GbE per sector         │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

## Apply

**Transport requirements summary:**

| Parameter | Requirement | Notes |
|-----------|-------------|-------|
| Bandwidth | 10-25 Gbps/RU | With compression |
| One-way latency | &lt;100 µs | Excludes DU/RU processing |
| Jitter | &lt;1 µs | For proper symbol timing |
| Phase sync | ±1.5 µs | At air interface |
| Frequency sync | ±50 ppb | SyncE required |
| Availability | 99.999% | Five nines |
| Max hops | 10-15 | PTP budget |

**Design checklist:**

```bash
# Bandwidth calculation
PRBs=273; LAYERS=4; SCS=30
BW_UNCOMPRESSED=$((PRBs * 12 * 2 * 32 * 14 * SCS / 15 * LAYERS / 1000000000))
echo "Uncompressed: ${BW_UNCOMPRESSED} Gbps"
# Compressed (BFP 9-bit): multiply by 0.6

# Latency budget (fiber)
FIBER_KM=10
LATENCY_US=$((FIBER_KM * 5))
echo "Fiber latency: ${LATENCY_US} µs"

# PTP verification
ptp4l -i eth0 -m
# Check offset < 100 ns

# SyncE verification
synce4l -i eth0 -m
# Check QL and SSM messages
```

**Switch requirements:**

| Feature | Purpose |
|---------|---------|
| 10/25 GbE ports | Bandwidth |
| PTP (G.8275.1) | Phase sync |
| SyncE | Frequency sync |
| 802.1Qbu | Frame preemption |
| 802.1Qbv | Time-aware shaper |
| Cut-through switching | Low latency |

<section class="lesson-pitfalls">

## Pitfalls

1. **Underestimating bandwidth.** Don't forget eCPRI overhead, multiple carriers, multiple sectors. Size for peak, not average.

2. **Ignoring jitter.** Latency is average. Jitter is variance. High jitter causes symbol timing errors even with low average latency.

3. **PTP through non-BC switches.** Every switch must be a boundary clock (T-BC). Transparent clocks (T-TC) add error beyond budget.

4. **SyncE disabled.** Without SyncE, PTP holdover during GNSS outage degrades quickly. Enable on every hop.

5. **Store-and-forward switches.** Cut-through switching reduces latency. Store-and-forward adds frame-size-dependent delay.

6. **VLAN QoS insufficient.** Priority queuing helps but doesn't guarantee determinism. TSN (Qbu/Qbv) provides bounded latency.

7. **Ring protection too slow.** G.8032 &lt;50 ms is fine for backhaul, not fronthaul. Consider &lt;1 ms protection or dual-homing.

8. **Distance miscalculation.** Fiber latency is 5 µs/km. Add switch latency per hop. Budget gets tight beyond 10-15 km.

</section>
