## Why this matters

eCPRI defines the transport. O-RAN WG4 defines what goes inside — the section structures, timing parameters, compression formats, and management interfaces.

Understanding the O-RAN fronthaul spec is essential for multi-vendor interoperability. This is where the interop details live.

</section>

## Intuition

### O-RAN fronthaul planes

```
O-RAN Open Fronthaul Planes:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  Four planes defined by O-RAN WG4:                      │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │ C-Plane (Control Plane)                         │    │
│  │ - Scheduling information                         │    │
│  │ - Section descriptions (PRB allocation)          │    │
│  │ - Beamforming information                        │    │
│  │ - Sent before U-Plane data                       │    │
│  │ Protocol: eCPRI message type 0x02               │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │ U-Plane (User Plane)                            │    │
│  │ - IQ sample data                                │    │
│  │ - Frequency domain (after DFT)                  │    │
│  │ - Compressed (BFP, modulation compression)      │    │
│  │ Protocol: eCPRI message type 0x00               │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │ S-Plane (Synchronization Plane)                 │    │
│  │ - Timing synchronization                         │    │
│  │ - Phase alignment                                │    │
│  │ Protocol: PTP (IEEE 1588), SyncE               │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │ M-Plane (Management Plane)                      │    │
│  │ - Configuration                                  │    │
│  │ - Performance monitoring                         │    │
│  │ - Fault management                               │    │
│  │ Protocol: NETCONF/YANG                          │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### CUS message flow

```
C-Plane and U-Plane Message Sequence:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  DU                                    RU                │
│  │                                     │                │
│  │ C-Plane (scheduling for slot N)     │                │
│  │ - Section type 1 (DL data)          │                │
│  │ - PRB allocation                     │                │
│  │ - Beam ID / weights                  │                │
│  │─────────────────────────────────────►│                │
│  │                                     │ Store C-Plane  │
│  │                                     │                │
│  │ U-Plane (IQ data for slot N)        │                │
│  │ - Symbol 0, PRBs 0-50               │                │
│  │─────────────────────────────────────►│                │
│  │                                     │                │
│  │ U-Plane (IQ data for slot N)        │                │
│  │ - Symbol 0, PRBs 51-100             │                │
│  │─────────────────────────────────────►│                │
│  │                                     │                │
│  │ ... more U-Plane messages ...       │                │
│  │─────────────────────────────────────►│                │
│  │                                     │                │
│  │                                     │ Apply C-Plane  │
│  │                                     │ to U-Plane     │
│  │                                     │                │
│  │                                     │ IFFT + TX      │
│  │                                     │─────────────►  │
│  │                                     │     Air        │
│                                                          │
│  Timing: C-Plane arrives T2a before slot boundary       │
│          U-Plane arrives Ta3 before symbol boundary     │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

## How it works

### C-Plane section structure

```
O-RAN C-Plane Section Format:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  Application Layer Header:                              │
│  ┌─────────────────────────────────────────────────┐    │
│  │ dataDirection (1 bit): 0=UL, 1=DL               │    │
│  │ payloadVersion (3 bits): 001 for current        │    │
│  │ filterIndex (4 bits): Identifies filter set     │    │
│  │ frameId (8 bits): Radio frame number mod 256    │    │
│  │ subframeId (4 bits): 0-9                        │    │
│  │ slotId (6 bits): Slot within subframe           │    │
│  │ startSymbolId (6 bits): Starting symbol         │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  Section Header (per section):                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │ sectionId (12 bits): Identifies section         │    │
│  │ rb (1 bit): Resource block indicator            │    │
│  │ symInc (1 bit): Symbol increment mode           │    │
│  │ startPrbc (10 bits): Starting PRB               │    │
│  │ numPrbc (8 bits): Number of PRBs (0=all)        │    │
│  │ reMask (12 bits): Which REs in PRB              │    │
│  │ numSymbol (4 bits): Number of symbols           │    │
│  │ ef (1 bit): Extension flag                      │    │
│  │ beamId (15 bits): Beam identifier               │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  Section Extensions (optional, ef=1):                   │
│  ┌─────────────────────────────────────────────────┐    │
│  │ extType 1: Beamforming weights                  │    │
│  │ extType 2: BFW compression parameters           │    │
│  │ extType 3: Time/frequency offset                │    │
│  │ extType 4: Modulation compression parameters    │    │
│  │ extType 5: Codebook index                       │    │
│  │ ...more extension types...                      │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### U-Plane IQ format

```
O-RAN U-Plane IQ Data Format:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  Application Layer Header:                              │
│  ┌─────────────────────────────────────────────────┐    │
│  │ dataDirection, payloadVersion, filterIndex      │    │
│  │ frameId, subframeId, slotId, symbolId           │    │
│  │ (Same as C-Plane header)                        │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  Section Header:                                        │
│  ┌─────────────────────────────────────────────────┐    │
│  │ sectionId: Must match C-Plane section           │    │
│  │ rb, symInc, startPrbc, numPrbc, reMask          │    │
│  │                                                  │    │
│  │ udCompHdr (if compression used):                │    │
│  │ - udIqWidth: IQ bit width (e.g., 9 bits)        │    │
│  │ - udCompMeth: Compression method                │    │
│  │   0: No compression                             │    │
│  │   1: Block floating point                       │    │
│  │   2: Block scaling                              │    │
│  │   3: µ-law                                      │    │
│  │   4: Modulation compression                     │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  IQ Sample Data (per PRB):                              │
│  ┌─────────────────────────────────────────────────┐    │
│  │ [exponent (4 bits if BFP)]                      │    │
│  │ [I0, Q0], [I1, Q1], ..., [I11, Q11]            │    │
│  │ (12 subcarriers per PRB)                        │    │
│  │                                                  │    │
│  │ BFP format (9-bit example):                     │    │
│  │ ┌─────────────────────────────────────────────┐ │    │
│  │ │ Exp │ I0 (9b) │ Q0 (9b) │ ... │ I11 │ Q11 │ │    │
│  │ │ 4b  │         │         │     │      │     │ │    │
│  │ └─────────────────────────────────────────────┘ │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### S-Plane timing

```
S-Plane Synchronization:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  PTP Profile: ITU-T G.8275.1                            │
│  ┌─────────────────────────────────────────────────┐    │
│  │ Full timing support (FTS)                       │    │
│  │ Every switch is a boundary clock               │    │
│  │ Target: ±1.5 µs at air interface               │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  Timing Chain:                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  GPS/GNSS ──► Grand Master ──► T-BC ──► T-BC   │    │
│  │  (source)     (DU/router)     (switch) (switch)│    │
│  │                                   │             │    │
│  │                                   ▼             │    │
│  │                                  RU             │    │
│  │                              (slave clock)      │    │
│  │                                                  │    │
│  │  Each hop adds error budget                     │    │
│  │  Total budget: ~1.5 µs for all hops            │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  SyncE for Frequency:                                   │
│  ┌─────────────────────────────────────────────────┐    │
│  │ Physical layer frequency sync                   │    │
│  │ Requirement: ±50 ppb (±16 ppb for wide area)   │    │
│  │ All switches must support and enable SyncE     │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

**NETCONF operations:**

```bash
# Get RU capabilities
netconf-cli get --filter "<o-ran-module-cap/>"

# Configure carrier
netconf-cli edit-config --target running <<EOF
<config>
  <user-plane-configuration xmlns="urn:o-ran:uplane-conf:1.0">
    <tx-array-carriers>
      <name>cc0</name>
      <absolute-frequency-center>3600000</absolute-frequency-center>
      <channel-bandwidth>100000</channel-bandwidth>
      <active>true</active>
    </tx-array-carriers>
  </user-plane-configuration>
</config>
EOF

# Check timing parameters
netconf-cli get --filter "<delay-management/>"

# View alarms
netconf-cli get --filter "<alarms/>"
```

<section class="lesson-pitfalls">

## Pitfalls

1. **Section ID mismatch.** U-Plane sectionId must match C-Plane. Mismatch causes data loss.

2. **Timing parameter incompatibility.** DU and RU must agree on T2a/Ta3 values. Negotiate via M-Plane.

3. **Compression mismatch.** udIqWidth and udCompMeth must match configuration. Wrong settings corrupt IQ data.

4. **YANG model version.** Different O-RAN spec versions have different YANG. Verify compatibility.

5. **M-Plane connectivity.** Hierarchical vs hybrid architecture must match deployment. Wrong model breaks management.

6. **Extension parsing.** Optional extensions (ef=1) must be parsed correctly. Unknown extensions should be skipped.

7. **PTP profile mismatch.** O-RAN requires G.8275.1. Wrong profile (e.g., G.8265.1) gives wrong accuracy.

8. **Symbol timing drift.** RU clock must stay synchronized. Drift causes symbol boundary misalignment.
