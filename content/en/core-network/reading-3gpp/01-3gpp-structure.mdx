## Why this matters

3GPP specifications are the authoritative source for mobile network standards. Understanding how to navigate them saves hours of searching and ensures you're reading the right document for your task.

### Reading tips

```
Practical Reading Tips:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  1. UNDERSTAND DOCUMENT STRUCTURE                       │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  Typical 3GPP spec layout:                      │    │
│  │  - Clause 1: Scope                              │    │
│  │  - Clause 2: References                         │    │
│  │  - Clause 3: Definitions and abbreviations      │    │
│  │  - Clause 4+: Technical content                 │    │
│  │  - Annexes: Informative examples                │    │
│  │                                                  │    │
│  │  Always read Clause 3 first!                    │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  2. FOLLOW REFERENCES                                   │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  "[X]" notation references Clause 2 list        │    │
│  │  "See TS 23.501 clause 5.4" → jump there        │    │
│  │                                                  │    │
│  │  Build a map of related specs as you read       │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  3. USE KEYWORD: "SHALL" vs "SHOULD" vs "MAY"           │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  SHALL  = mandatory, must implement             │    │
│  │  SHOULD = recommended, expect most will         │    │
│  │  MAY    = optional, up to implementation        │    │
│  │                                                  │    │
│  │  Search for "shall" to find requirements        │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  4. CHECK VERSION/RELEASE                               │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  - Pick release matching your deployment        │    │
│  │  - Latest isn't always best (stability)         │    │
│  │  - Use version comparison for changes           │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

## Apply

**Quick reference URLs:**

| Resource | URL |
|----------|-----|
| Spec finder | https://portal.3gpp.org/desktopmodules/Specifications/SpecificationDetails.aspx |
| Spec series | https://www.3gpp.org/specifications/specification-numbering |
| Release info | https://www.3gpp.org/specifications/releases |
| Work plan | https://www.3gpp.org/specifications/work-plan |

**Command-line navigation:**

```bash
# Download spec from 3GPP (after finding URL)
wget "https://www.3gpp.org/ftp/Specs/archive/23_series/23.501/23501-h60.zip"
unzip 23501-h60.zip

# Search within spec (after extracting)
grep -i "registration" TS_23.501_*.docx 2>/dev/null || \
  # If docx, use alternative tool
  unzip -p TS_23.501*.docx word/document.xml | grep -i "registration"

# Compare versions (using diff on text exports)
diff ts23501_v16.txt ts23501_v17.txt | head -100
```

**Common lookup patterns:**

```bash
# Finding spec for a topic:

1. "How does UE registration work?"
   → Start: TS 23.502 (procedures)
   → Section: 4.2.2 Registration

2. "What is the AMF API for registration?"
   → Start: TS 29.518 (AMF services)
   → Look for Namf_Communication

3. "What are the NAS messages for registration?"
   → Start: TS 24.501 (NAS protocol)
   → Section: 8 (message definitions)

4. "How is registration secured?"
   → Start: TS 33.501 (security)
   → Section: 6 (authentication)
```

<section class="lesson-pitfalls">

## Pitfalls

1. **Wrong release.** Reading Rel-15 spec for Rel-17 feature. Check release first.

2. **Stage confusion.** Stage 1 (services) ≠ Stage 2 (architecture) ≠ Stage 3 (protocols). Know which you need.

3. **Abbreviation overload.** 3GPP loves abbreviations. Keep Clause 3 open at all times.

4. **Circular references.** Spec A references B which references A. Build a mental map.

5. **"FFS" trap.** "For Further Study" means not yet defined. Don't implement FFS items.

6. **Outdated local copy.** Using a downloaded spec from years ago. Check for updates.

7. **Normative vs informative.** Annexes are often informative (examples). Main body is normative.

8. **Missing context.** Reading TS 29.xxx without understanding TS 23.xxx architecture. Start with architecture.
