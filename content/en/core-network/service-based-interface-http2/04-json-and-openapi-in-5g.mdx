## Why this matters

5G SBI APIs aren't documented in prose — they're defined in machine-readable OpenAPI specifications. Every request body, response, and error is formally specified.

This means you can generate client code, validate messages, and debug interoperability issues by reading the spec. Understanding OpenAPI in 5G context is essential for implementation and troubleshooting.

</section>

## Intuition

### OpenAPI as the contract

```
OpenAPI in 5G Ecosystem:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  3GPP Specification (TS 29.xxx)                         │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  Prose Description          OpenAPI YAML        │    │
│  │  (normative text)           (machine-readable)  │    │
│  │        │                         │              │    │
│  │        └─────────┬───────────────┘              │    │
│  │                  │                              │    │
│  │                  ▼                              │    │
│  │  ┌──────────────────────────────────────┐      │    │
│  │  │ Both are normative!                  │      │    │
│  │  │ OpenAPI is the source of truth for   │      │    │
│  │  │ API structure, types, and validation │      │    │
│  │  └──────────────────────────────────────┘      │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  Vendors generate code from OpenAPI → interoperability  │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### JSON over alternatives

```
Why JSON for 5G SBI:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  Alternative      │ Why Not Used                        │
│  ─────────────────┼────────────────────────────────────│
│  ASN.1 (4G)       │ Complex, requires special tools    │
│  Protocol Buffers │ Binary, less human-readable        │
│  XML              │ Verbose, parsing overhead          │
│  YAML             │ Superset of JSON, no added value   │
│                                                          │
│  JSON Advantages:                                       │
│  ├── Human-readable for debugging                      │
│  ├── Native to HTTP/REST ecosystem                     │
│  ├── Wide tooling support                              │
│  ├── HPACK compresses repetitive headers              │
│  └── Simple type system matches telecom needs         │
│                                                          │
│  Trade-off: JSON is ~30-50% larger than binary formats │
│  but HTTP/2 compression + HPACK offsets this          │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

## How it works

### OpenAPI structure for 5G

```
Typical 5G API OpenAPI Structure:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  openapi: 3.0.0                                         │
│  info:                                                  │
│    title: Nsmf_PDUSession                              │
│    version: 1.0.0                                      │
│                                                          │
│  servers:                                               │
│    - url: '{apiRoot}/nsmf-pdusession/v1'              │
│                                                          │
│  paths:                                                 │
│    /sm-contexts:                                       │
│      post:                                             │
│        operationId: CreateSMContext                    │
│        requestBody:                                    │
│          content:                                      │
│            application/json:                           │
│              schema:                                   │
│                $ref: '#/components/schemas/...'       │
│        responses:                                      │
│          '201':                                        │
│            description: Created                        │
│          '400':                                        │
│            $ref: 'TS29571_CommonData.yaml#/...'       │
│                                                          │
│  components:                                            │
│    schemas:                                            │
│      SmContextCreateData:                              │
│        type: object                                    │
│        required:                                       │
│          - supi                                        │
│          - pduSessionId                               │
│        properties:                                     │
│          supi:                                         │
│            $ref: 'TS29571_CommonData.yaml#/.../Supi'  │
│          pduSessionId:                                │
│            $ref: 'TS29571_CommonData.yaml#/.../...'   │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

## Apply

**Finding OpenAPI specs:**

| Source | Location |
|--------|----------|
| 3GPP official | https://www.3gpp.org/ftp/Specs/archive/ |
| API directory | TS 29.5xx series (each NF has own spec) |
| Common types | TS 29.571 |
| Format | YAML files in spec archives |

**Validating messages:**

```bash
# Generate types from OpenAPI
openapi-generator generate -i TS29502_Nsmf.yaml -g python

# Validate JSON against schema
ajv validate -s schema.json -d message.json

# Wireshark: view JSON bodies
http2.data && json

# Pretty-print JSON from logs
jq '.' /var/log/smf/sbi-trace.json
```

**Common validation patterns:**

| Field | Validation | Regex/Pattern |
|-------|------------|---------------|
| SUPI | imsi-\{15 digits\} | `^imsi-[0-9]{15}$` |
| GPSI | msisdn-\{digits\} | `^msisdn-[0-9]{5,15}$` |
| DNN | ASCII string | `^[a-zA-Z0-9.-]+$` |
| S-NSSAI SST | 0-255 | integer |
| S-NSSAI SD | 6 hex digits | `^[0-9A-Fa-f]{6}$` |

<section class="lesson-pitfalls">

## Pitfalls

1. **Spec version mismatch.** OpenAPI specs evolve with 3GPP releases. Ensure you're using the right version (R15, R16, R17).

2. **Unknown fields handling.** Some implementations reject unknown JSON fields. Others ignore them. Test interop carefully.

3. **Null vs absent.** JSON null is different from absent field. Some APIs are strict about this.

4. **Base64 encoding.** Binary data must be base64-encoded correctly. Padding issues are common.

5. **DateTime format.** Must be RFC 3339 (ISO 8601). Time zone required. "Z" for UTC.

6. **$ref resolution.** Cross-file references require all referenced specs to be available.

7. **Multipart parsing.** Boundary handling is tricky. Content-Id must match exactly.

8. **Enum case sensitivity.** 3GPP enums are usually UPPER_CASE. "registered" ≠ "REGISTERED".
