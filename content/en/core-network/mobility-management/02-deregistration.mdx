## Why this matters

Registration gets a UE onto the network. **Deregistration** takes it off. This happens when you power off your phone, airplane mode activates, or the network decides to disconnect you.

Understanding deregistration helps you debug "disappeared device" scenarios and optimize network resource management.

</section>

## Intuition

### When deregistration happens

```
Deregistration Triggers:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  UE-Initiated:                                          │
│  ├── Power off / shutdown                               │
│  ├── Airplane mode                                      │
│  ├── SIM removal                                        │
│  └── Switch off 5G (settings change)                   │
│                                                          │
│  Network-Initiated:                                     │
│  ├── Subscription terminated                            │
│  ├── OAM administrative action                          │
│  ├── Re-auth failure                                    │
│  └── Illegal UE detection                               │
│                                                          │
│  Implicit (automatic):                                  │
│  ├── Periodic registration timer (T3512) expiry        │
│  ├── Mobile reachable timer expiry                     │
│  └── Implicit deregistration timer expiry              │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### Deregistration types

| Type | Initiator | Example |
|------|-----------|---------|
| **Explicit UE** | UE | User powers off phone |
| **Explicit Network** | AMF | Admin disconnects user |
| **Implicit** | Timer | UE went silent too long |

```

┌─────────────────────────────────────────────────────────┐
```

```
│  │ AMF sends Nsmf_PDUSession_ReleaseSMContext      │    │
│  │ to each SMF for all PDU sessions                │    │
```

## Apply

**Timer configuration:**

| Timer | Purpose | Typical Value |
|-------|---------|---------------|
| T3512 | Periodic registration | 54 min |
| Mobile reachable | Start paging after T3512 miss | 4 min |
| Implicit dereg | Release context if unreachable | 4 min |
| T3522 | Deregistration retry | 6 sec |

**Debugging deregistration:**

```bash
# Wireshark filter for deregistration
nas-5gs.mm.message_type == 0x45 || nas-5gs.mm.message_type == 0x46

# Check UDM registration status
curl -X GET "https://udm.5gc/nudm-uecm/v1/imsi-310260123456789/registrations"
```

**Common scenarios:**

| Symptom | Check |
|---------|-------|
| UE can't reconnect after power cycle | Stale context in AMF, wait for implicit dereg |
| Repeated deregistration | Network-initiated cause, check AMF logs |
| Sessions not released | SMF notified? Check N11 interface |
| UDM still shows registered | UECM deregistration failed, check SBI |

<section class="lesson-pitfalls">

## Pitfalls

1. **Forgetting switch-off flag.** When UE powers off, it sets switch-off=1 so network doesn't wait for accept. Missing this wastes time.

2. **Not releasing PDU sessions.** AMF must notify SMF to release sessions. Missing this leaks resources.

3. **Ignoring implicit deregistration.** If UE disappears (battery dies, out of coverage), implicit dereg is the only cleanup path.

4. **Too aggressive timers.** Very short implicit dereg timers cause frequent re-registrations for UEs in bad coverage.

5. **Not updating UDM.** AMF must call Nudm_UECM_Deregistration. Missing this leaves stale registration records.

6. **Re-registration loop.** If deregistration cause requires re-registration but underlying issue isn't fixed, UE loops.

7. **Context mismatch after failure.** If AMF crashes mid-deregistration, UE and network state diverge. Need recovery procedures.

8. **Multi-access confusion.** UE can be registered over 3GPP and non-3GPP simultaneously. Deregistration must specify which access.
