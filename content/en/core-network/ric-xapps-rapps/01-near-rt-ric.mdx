## Why this matters

Traditional RAN operates with static algorithms. Near-RT RIC enables real-time optimization — making decisions in 10ms to 1s based on live network conditions.

This is where AI/ML meets the RAN. Understanding Near-RT RIC is essential for modern O-RAN deployments.

## Apply

**E2AP message types:**

| Message | Direction | Purpose |
|---------|-----------|---------|
| E2 Setup Request | E2 Node → RIC | Initial connection |
| E2 Setup Response | RIC → E2 Node | Accept connection |
| RIC Subscription | RIC → E2 Node | Subscribe to data |
| RIC Indication | E2 Node → RIC | Report data |
| RIC Control | RIC → E2 Node | Send action |
| E2 Node Config Update | E2 Node → RIC | Capability change |

**OSC Near-RT RIC deployment:**

```bash
# Deploy OSC Near-RT RIC (Kubernetes)
helm repo add oran-sc https://gerrit.o-ran-sc.org/r/ric-plt/ric-dep
helm install ric oran-sc/ric-platform

# Check RIC pods
kubectl get pods -n ricplt

# View E2 connections
kubectl logs -n ricplt -l app=e2term | grep "E2 Setup"

# Deploy an xApp
curl -X POST http://appmgr:8080/ric/v1/xapps \
  -d '{"xappName": "my-xapp", "helmVersion": "1.0.0"}'
```

<section class="lesson-pitfalls">

## Pitfalls

1. **E2 latency exceeded.** Control loop must complete in &lt;1s. Network delay + processing must fit budget.

2. **Subscription overload.** Too many xApps subscribing to same data. Use subscription merging.

3. **Conflicting xApp actions.** MLB xApp and MRO xApp issue conflicting commands. Enable conflict mitigation.

4. **E2SM version mismatch.** E2 node and xApp use different E2SM versions. Check compatibility.

5. **SDL bottleneck.** All xApps hammering Redis. Optimize read/write patterns, use local cache.

6. **Missing RAN function.** xApp subscribes to function E2 node doesn't support. Check E2 Setup response.

7. **Control action rejected.** E2 node refuses action due to constraints. Handle RIC Control Failure.

8. **xApp crash recovery.** xApp restarts, loses subscription state. Use subscription manager for persistence.
