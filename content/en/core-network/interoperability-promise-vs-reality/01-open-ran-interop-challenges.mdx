## Why this matters

Open RAN promises multi-vendor interoperability, but "open interfaces" doesn't guarantee seamless integration. Understanding the real challenges helps set realistic expectations and plan accordingly.

</section>

<section class="lesson-intuition">

## Intuition

### The interop gap

```
Expectation vs Reality:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  EXPECTATION                                            │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  RU Vendor A ──┬── O-RAN FH ──┬── DU Vendor X   │    │
│  │  RU Vendor B ──┤              ├── DU Vendor Y   │    │
│  │  RU Vendor C ──┘              └── DU Vendor Z   │    │
│  │                                                  │    │
│  │  "Any RU works with any DU"                     │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  REALITY                                                │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  RU Vendor A ── tested ── DU Vendor X    ✓      │    │
│  │  RU Vendor A ── ? ── DU Vendor Y         ?      │    │
│  │  RU Vendor B ── incompatible ── DU Vendor X  ✗  │    │
│  │  RU Vendor C ── not tested ── DU Vendor Z   ?   │    │
│  │                                                  │    │
│  │  "Specific combinations validated"              │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  Gap reasons:                                           │
│  - Spec ambiguity → different interpretations          │
│  - Optional features → incompatible choices            │
│  - Timing parameters → different defaults              │
│  - Profiles → vendor-specific subsets                  │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### Specification challenges

```
Why Specs Don't Guarantee Interop:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  1. OPTIONAL FEATURES                                   │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  O-RAN Fronthaul spec defines:                  │    │
│  │  - Mandatory features (M)                       │    │
│  │  - Optional features (O)                        │    │
│  │  - Conditional features (C)                     │    │
│  │                                                  │    │
│  │  Vendor A implements: M + O1 + O3               │    │
│  │  Vendor B implements: M + O2 + O4               │    │
│  │                                                  │    │
│  │  Both are "compliant" but incompatible on O     │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  2. PARAMETER RANGES                                    │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  Spec says: "T2a min: 25µs to 500µs"            │    │
│  │                                                  │    │
│  │  RU Vendor: implements 50µs                     │    │
│  │  DU Vendor: expects 100µs minimum               │    │
│  │                                                  │    │
│  │  Both compliant, but won't work together        │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  3. ENCODING CHOICES                                    │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  IQ sample format: spec allows multiple         │    │
│  │  - Block floating point                         │    │
│  │  - Static compression                           │    │
│  │  - μ-law compression                            │    │
│  │                                                  │    │
│  │  If RU uses BFP and DU expects static:          │    │
│  │  → Data flows but decodes incorrectly           │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  4. PROFILES                                            │
│  ┌─────────────────────────────────────────────────┐    │
│  │                                                  │    │
│  │  O-RAN WG4 defines capability profiles:         │    │
│  │  - Profile A: minimal                           │    │
│  │  - Profile B: mid-range                         │    │
│  │  - Profile C: full-featured                     │    │
│  │                                                  │    │
│  │  RU Profile A + DU Profile C:                   │    │
│  │  → DU expects features RU doesn't have          │    │
│  │                                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

## Apply

**Integration checklist:**

| Interface | Key Verification | Tool/Method |
|-----------|-----------------|-------------|
| Fronthaul | Timing alignment | Protocol analyzer |
| Fronthaul | IQ sample format | Capture and decode |
| E2 | Service model match | ASN.1 comparison |
| O1 | YANG compatibility | YANG validator |
| A1 | Policy schema | JSON schema check |

**Debug commands:**

```bash
# Fronthaul timing check
tcpdump -i eth0 -w fh_capture.pcap port 44944
# Analyze with Wireshark O-RAN dissector

# PTP sync verification
pmc -u -b 0 'GET TIME_STATUS_NP'
# Check offset from master < 100ns

# E2 connection status
curl http://e2mgr:3800/v1/nodeb/states
# All E2 nodes should show "CONNECTED"

# O1 NETCONF test
netconf-console --host=odu --port=830 --get-config
# Should return valid configuration

# YANG model diff
pyang --check-update-from=prev.yang current.yang
# Identify breaking changes
```

<section class="lesson-pitfalls">

## Pitfalls

1. **"O-RAN Compliant" confusion.** Compliant = self-declared. Certified = tested. Demand certification.

2. **Version lock.** RU firmware v1.2 tested with DU v3.4. Any update may break. Test every upgrade.

3. **Lab vs field.** Lab integration succeeds but field has different timing, interference. Budget for field iteration.

4. **Finger-pointing.** Multi-vendor = "not my problem." Contractually assign integration responsibility.

5. **Performance delta.** Multi-vendor may have 5-15% lower throughput initially. Plan for optimization time.

6. **Support escalation.** Vendor A blames Vendor B. Require joint debugging in contracts.

7. **Feature roadmap mismatch.** DU adds feature, RU doesn't support. Coordinate release planning.

8. **Operator-specific extensions.** Adding custom features breaks interop. Minimize proprietary extensions.
