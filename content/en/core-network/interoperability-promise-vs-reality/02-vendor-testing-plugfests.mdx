## Why this matters

Plugfests are where multi-vendor interoperability gets proven (or disproven). Understanding how these testing events work helps operators evaluate vendor claims and plan integration strategies.

</section>

<section class="lesson-apply">

## Apply

**Evaluating vendor interop claims:**

| Claim | Verification |
|-------|--------------|
| "O-RAN Compliant" | Self-declared, no testing. Ask for OTIC certification. |
| "Tested at PlugFest" | Check O-RAN TIFG reports. Which interface? What results? |
| "O-RAN Certified" | Verify on O-RAN Alliance certified products list |
| "Interop tested with X" | Request test report. Bilateral or operator-witnessed? |
| "Production deployed" | Reference customer. Similar use case? |

**RFP interoperability requirements:**

```bash
# Mandatory requirements for RFP:

1. Certification
   "Product must be O-RAN Certified (Silver or Gold level)
    for all interfaces being procured"

2. PlugFest participation
   "Vendor must demonstrate participation in at least
    2 O-RAN Global PlugFests within last 18 months"

3. Interop evidence
   "Provide test reports showing interoperability with
    at least 3 other vendors' products per interface"

4. Operator lab commitment
   "Vendor agrees to participate in operator integration
    lab testing for minimum 4 weeks"

5. Field trial support
   "Vendor provides on-site engineering support during
    field trial phase (minimum 3 months)"

6. Regression testing
   "Vendor commits to regression testing for all
    software updates before deployment"
```

**Key questions for vendors:**

```bash
# During vendor evaluation:

Q1: "What specific interfaces are certified?"
    → Fronthaul only? E2? O1? All?

Q2: "Which test cases passed/failed at last PlugFest?"
    → Request detailed results, not just participation

Q3: "What firmware/software version was certified?"
    → Current shipping version or older?

Q4: "What other vendors have you integrated with?"
    → In lab? In production? Same configuration?

Q5: "What's your regression testing process?"
    → How do you ensure updates don't break interop?

Q6: "Will you participate in joint debugging?"
    → Contractual commitment to multi-vendor support?
```

<section class="lesson-pitfalls">

## Pitfalls

1. **PlugFest ≠ Production.** Lab conditions differ from real networks. PlugFest success is necessary but not sufficient.

2. **Version mismatch.** Certified version may differ from shipping version. Verify exact version tested.

3. **Partial certification.** Vendor certified for fronthaul but not E2. Check each interface separately.

4. **Bilateral vs multilateral.** Tested with one vendor doesn't mean works with all. Check specific combinations.

5. **Test scope.** Basic attach tested but not handover. Understand what scenarios were validated.

6. **Time decay.** Certification from 2 years ago may not reflect current product. Check certification date.

7. **Profile mismatch.** Certified at Profile A but you need Profile C features. Verify capability profiles.

8. **Lab environment.** PlugFest uses ideal conditions. Real networks have interference, timing variations.
