## Why this matters

"This router is reliable" is subjective. "This AMF has a hardware MTBF of 1,000,000 hours but a software MTBF of 200 hours due to memory leaks" is actionable.

MTBF and MTTR are the foundational metrics for predicting and measuring reliability. They are used to design redundancy, calculate availability, compare equipment, and plan maintenance. Every telecom engineer needs to understand them — and cloud engineers need to understand how the classical hardware model extends (and where it breaks) when applied to containerized 5G Network Functions.

</section>

<section class="lesson-intuition">

## Intuition

### What is MTBF?

**MTBF (Mean Time Between Failures)** is the average time a system operates before failing.

```
         Failure 1         Failure 2         Failure 3
            │                  │                  │
────────────┼──────────────────┼──────────────────┼────────────
            │                  │                  │
      Time A = 1000h     Time B = 1200h     Time C = 800h

MTBF = (1000 + 1200 + 800) / 3 = 1000 hours
```

**Higher MTBF = More reliable**

A carrier-grade router with MTBF of 1,000,000 hours is statistically expected to fail once every ~114 years. But this is a statistical average, not a guarantee — any individual unit may fail much sooner.

### What is MTTR?

**MTTR (Mean Time To Repair)** is the average time to restore a failed system to operation.

```
Failure occurs        System restored
      │                     │
──────┼─────────────────────┼──────────────────────
      │                     │
      │◄──── MTTR = 4h ────►│
      │                     │
  Detection + Diagnosis + Repair + Verification
```

**Lower MTTR = Faster recovery**

MTTR includes:
- **Detection time:** How long until you know it failed?
- **Diagnosis time:** What failed and why?
- **Repair time:** Fix or replace
- **Verification time:** Confirm it is working

### The availability equation

```
Availability = MTBF / (MTBF + MTTR)
```

**Example:**
```
MTBF = 10,000 hours
MTTR = 1 hour

Availability = 10,000 / (10,000 + 1) = 99.99%
```

**Key insight:** You can improve availability by:
- Increasing MTBF (more reliable components)
- Decreasing MTTR (faster repair)
- Adding redundancy (masks failures entirely)

### Hardware vs software MTBF

Here is where cloud engineers need a mental model shift:

| Failure type | MTBF range | Cause | MTTR |
|-------------|------------|-------|------|
| Hardware (NIC, ASIC) | 500K–1M hours | Physical wear, ESD | Hours (swap) |
| OS / hypervisor | 10K–100K hours | Kernel panic, resource leak | Minutes (reboot) |
| Container / pod | 100–10K hours | OOM kill, crash loop, config error | Seconds–minutes (restart) |
| Software bug (specific trigger) | Unpredictable | Edge case in traffic pattern | Seconds (pod restart) |

A 5G Network Function running in Kubernetes may have hardware MTBF of 1,000,000 hours but software MTBF of 500 hours if the NF process has a memory leak that causes OOM kill every three weeks. The hardware is irrelevant — the software is the bottleneck.

<section class="lesson-mechanism">

## How it works

### Calculating MTBF

**For a single component:**

```
MTBF = Total operating time / Number of failures

Example:
100 routers run for 1 year (8,760 hours)
5 failures total

MTBF = (100 × 8,760) / 5 = 175,200 hours
```

**For a system with components in series:**

If any component fails, the system fails (no redundancy).

```
┌────────┐    ┌────────┐    ┌────────┐
│ Comp A │───►│ Comp B │───►│ Comp C │
│MTBF=10K│    │MTBF=20K│    │MTBF=50K│
└────────┘    └────────┘    └────────┘

1/MTBF_system = 1/MTBF_A + 1/MTBF_B + 1/MTBF_C
1/MTBF_system = 1/10,000 + 1/20,000 + 1/50,000
MTBF_system = 5,882 hours
```

The system MTBF is always **lower** than any individual component in a series chain.

**For components in parallel (redundant):**

The system fails only when all components fail simultaneously.

```
┌────────┐
│ Comp A │──┐
│MTBF=10K│  │
└────────┘  ├───► Output
┌────────┐  │
│ Comp B │──┘
│MTBF=10K│
└────────┘

For identical active-standby components:
MTBF_system ≈ MTBF² / (2 × MTTR)   (with repair during operation)

With MTBF=10,000h, MTTR=4h:
MTBF_system ≈ 10,000² / (2 × 4) = 12,500,000 hours
```

Redundancy dramatically extends effective MTBF by changing the failure condition from "either fails" to "both fail before one is repaired."

### Components of MTTR

**MTTR breakdown:**

| Phase | Typical time | How to reduce |
|-------|--------------|---------------|
| Detection | Minutes to hours | Better monitoring, shorter probe intervals |
| Escalation | Minutes | Clear on-call procedures |
| Diagnosis | Minutes to hours | Better logging, structured runbooks |
| Repair | Seconds–hours | Auto-restart (software), spares (hardware) |
| Verification | Seconds–minutes | Automated health checks |

**Example MTTR calculation for hardware failure:**

```
Detection:     15 minutes (monitoring alerted)
Escalation:    10 minutes (NOC contacted engineer)
Diagnosis:     30 minutes (identified failed line card)
Repair:        90 minutes (swap line card, config restore)
Verification:  15 minutes (run diagnostics, traffic test)

MTTR = 160 minutes = 2.67 hours
```

**Example MTTR for software failure (containerized NF):**

```
Detection:      30 seconds (liveness probe failure → Kubernetes restarts pod)
Escalation:     0 (automated)
Repair:         90 seconds (pod restart, readiness probe passes)
Verification:   30 seconds (health check, traffic resumes)

MTTR = ~2.5 minutes
```

This is why cloud-native 5G NFs can achieve very low MTTR for software failures — the repair is automated. The challenge shifts to **ensuring the pod restart actually fixes the problem** (not just hiding it by restarting into the same broken state).

### The bathtub curve

Components follow a characteristic failure rate pattern across their lifetime:

```
Failure
Rate
  │
  │  Infant          Useful life           Wear-out
  │  mortality       (constant rate)       (increasing)
  │    │                  │                    │
  │  ──┘                  │                ┌───│
  │    ╲             ─────┼────────────────┤   │
  │     ╲                 │                │  ╱
  │      ╲────────────────┴────────────────┴─╱
  │
  └──────────────────────────────────────────────► Time
```

**Three phases:**
1. **Infant mortality:** Early failures from manufacturing defects, solder joint issues, inadequate QA. Caught by burn-in testing before deployment.
2. **Useful life:** Constant, random failure rate. MTBF is defined for this phase. This is the phase where the Poisson process model applies.
3. **Wear-out:** Increasing failure rate as capacitors dry, solder joints fatigue, and mechanical parts wear. Replace before this phase.

**Software equivalent of the bathtub curve:**

Software does not wear out physically, but it has analogous phases:
- Early bugs found in QA and initial deployment
- Stable operation with rare crashes
- "Bit rot" as dependencies age, certificate expiry, config drift accumulate

### Availability calculations

**Single system:**

```
Availability = MTBF / (MTBF + MTTR)

Example:
MTBF = 50,000 hours
MTTR = 4 hours
Availability = 50,000 / 50,004 = 99.992%
```

**Redundant systems (active-standby):**

```
Availability_combined = 1 - (Unavailability_A × Unavailability_B)

Where: Unavailability = MTTR / (MTBF + MTTR)

Example:
Each system: MTBF = 50,000h, MTTR = 4h
Unavailability = 4 / 50,004 = 0.00008

Combined unavailability = 0.00008 × 0.00008 = 6.4 × 10^-9
Availability = 1 - 6.4 × 10^-9 ≈ 99.9999994%
```

This assumes the failures are **independent** — a critical assumption discussed in the pitfalls section.

### 5G-specific MTBF/MTTR considerations

**Software MTBF for 5G Network Functions:**

5G NFs are microservices. Their failure modes differ from hardware:

```
AMF failure modes:
├── Process crash (SIGSEGV, OOM)   → MTTR: pod restart ~2 min
├── Deadlock / stuck              → MTTR: liveness probe restart ~5 min
├── Config error on deploy        → MTTR: rollback ~10 min
├── Database connection lost      → MTTR: reconnect ~1 min
└── Cascading overload (from SMF) → MTTR: hours if not detected early

UPF failure modes:
├── DPDK memory exhaustion        → MTTR: pod restart ~3 min
├── GTP-U table full              → MTTR: session cleanup ~5 min
└── Kernel bypass driver crash    → MTTR: node reboot ~15 min (worse)
```

**ISSU (In-Service Software Upgrade) and MTTR:**

Rolling upgrades are a key MTTR-reduction mechanism in 5G:

```
Traditional upgrade (planned downtime):
  Stop NF → Upgrade → Restart → Verify
  MTTR = ~30 min (scheduled)

ISSU rolling upgrade:
  Upgrade pod 1 (pod 2 handles traffic) →
  Wait for readiness probe →
  Upgrade pod 2 →
  MTTR = 0 (no user-visible outage)
```

ISSU requires stateless NF design (TS 23.501 clause 4.2) — session state stored in UDR, not local memory. Without stateless design, upgrading a pod kills all active sessions on that pod.

## Apply

### Comparing equipment

| Router class | MTBF | Cost tier | Best for |
|-------------|------|-----------|----------|
| Entry-level | 100,000h | $ | Non-critical branch |
| Enterprise | 300,000h | $$ | Business services |
| Carrier-grade | 1,000,000h | $$$ | Telecom core |

For software: MTBF is not a vendor spec — you measure it from your own incident history.

### MTTR reduction strategies

| Strategy | What it reduces | Cost |
|----------|----------------|------|
| Better monitoring (shorter probe intervals) | Detection time | Low |
| Liveness + startup probes (Kubernetes) | Detection + repair (automated) | Low |
| On-site spares for hardware | Repair time | Medium |
| Trained staff 24/7 | All phases | High |
| Hot-standby with failover automation | MTTR near zero | High |
| ISSU / rolling upgrade | Upgrade MTTR to zero | Design investment |

### Designing for availability

```
Target: 99.99% (52.6 minutes downtime/year)

Option 1: Single highly reliable system
  MTBF needed = MTTR × 9,999
  If MTTR = 4h, MTBF must be 40,000h
  Single router at 200,000h MTBF: 99.998% — close but single point of failure

Option 2: Redundant systems
  Two routers, each MTBF = 200,000h, MTTR = 4h
  Combined availability ≈ 99.9999996%

Redundancy almost always wins over ultra-reliable single systems.
```

### Tracking actual MTBF/MTTR

Design uses predicted MTBF/MTTR from vendor specs or reliability models. Operations must track actual values from incidents:

```
Actual MTBF = Total operating hours / Incident count (rolling 12 months)
Actual MTTR = Sum of all incident durations / Incident count

If actual MTBF << predicted MTBF:
  → Software bug, config issue, environment mismatch

If actual MTTR >> planned MTTR:
  → Detection is slow (monitoring gap), diagnosis is slow (missing logs),
    repair is slow (no spare parts, no runbook)
```

Reviewing actual vs predicted quarterly reveals where investment will improve availability most.

### Practical example: Core AMF deployment

```
Requirement: 99.9999% availability for AMF (handles mobility for 10M subscribers)

Given:
- AMF pod MTBF (hardware): 500,000 hours
- AMF software MTBF: 2,000 hours (one crash every ~83 days per pod)
- MTTR hardware: 4 hours
- MTTR software: 3 minutes (pod restart)

Single AMF pod:
  Hardware: 500,000 / 500,004 = 99.9992%
  Software:   2,000 / 2,000.05 = 99.9975%
  Combined single pod: ~99.997%  ← software dominates, not hardware

Two AMF pods (active-active, N+1):
  Software unavailability per pod = 3min / (2000h × 60) = 2.5 × 10^-5
  Combined (both fail simultaneously): (2.5 × 10^-5)² = 6.25 × 10^-10
  Availability ≈ 99.9999994%  ✓
```

The lesson: for cloud-native NFs, **software MTBF dominates**. Measure and improve it — hardware specs are secondary.

<section class="lesson-pitfalls">

## Pitfalls

1. **Confusing MTBF with guaranteed lifespan.** MTBF of 500,000 hours does not mean the device lasts 57 years. It means the *average* inter-failure interval across a large population. Individual units can fail on day one.

2. **Ignoring software MTBF.** For cloud-native NFs, software failures dominate hardware failures by orders of magnitude. An AMF pod with MTBF of 2,000 hours is the reliability bottleneck, not the 1,000,000-hour hardware.

3. **Assuming failure independence in redundant systems.** The redundancy availability formula assumes A and B fail independently. If both AMF pods are on the same Kubernetes node, a node failure takes both. If both run the same software version, a software bug crashes both simultaneously. These are common-cause failures and they destroy the independence assumption.

4. **Optimistic MTTR assumptions.** "3 minutes for pod restart" assumes Kubernetes is healthy, the image is cached, the readiness probe works correctly, and no pending configuration error. Real-world MTTR often exceeds estimates significantly.

5. **Not counting detection time in MTTR.** MTTR starts from failure, not from alert. If your monitoring polls every 5 minutes, detection alone adds up to 5 minutes before repair begins. For URLLC services with 1ms latency SLA, even detection time matters — use BFD (Bidirectional Forwarding Detection) for sub-second failure detection.

6. **Manufacturer MTBF optimism.** Vendors calculate MTBF under ideal lab conditions. Actual field MTBF is typically 30–50% of the vendor specification in real deployment conditions (temperature variation, power quality, vibration).

7. **Ignoring the bathtub curve tails.** MTBF applies to the useful-life flat portion. Brand-new hardware and aging hardware have higher failure rates. Run burn-in testing for new deployments and plan replacement before wear-out begins.

8. **Crash-loop masking real MTBF.** If a pod crashes and Kubernetes restarts it every 5 minutes, your incident log shows "pod restarted" but MTBF looks like 5 minutes. The root cause (memory leak, config bug, dependency failure) must be fixed — not just accepted as "normal pod churn."
