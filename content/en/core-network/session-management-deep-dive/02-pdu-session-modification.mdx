## Why this matters

Once a PDU session is established, it doesn't stay static. You start a video call — network adds a QoS flow. You hit a data cap — network throttles your AMBR. You move to an edge cell — UPF changes.

**Session modification** handles all these runtime changes without tearing down and rebuilding the session.

</section>

<section class="lesson-intuition">

## Intuition

### What can be modified

```
Modifiable Session Parameters:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  QoS Flows:                                             │
│  ├── Add new QoS flow (e.g., for video call)           │
│  ├── Delete QoS flow (call ended)                       │
│  └── Modify QoS parameters (5QI, MBR, GBR)             │
│                                                          │
│  Session AMBR:                                          │
│  ├── Increase (upgraded plan, off-peak hours)          │
│  └── Decrease (data cap reached, congestion)           │
│                                                          │
│  UPF/Data Path:                                         │
│  ├── Change UPF (SSC mode 2/3, mobility)               │
│  └── Add/remove UL classifier (edge breakout)          │
│                                                          │
│  QoS Rules:                                             │
│  ├── Add packet filter rule                            │
│  └── Modify/delete existing rules                      │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### Who triggers modification

| Initiator | Trigger Example |
|-----------|-----------------|
| **UE** | Start video call, request higher QoS |
| **SMF** | Policy change from PCF |
| **AMF** | Handover requires path update |
| **AF** | External app requests QoS via NEF |

<section class="lesson-apply">

## Apply

**Common modification triggers:**

| Trigger | What Changes |
|---------|--------------|
| Video call start | Add GBR QoS flow (5QI=2) |
| Video call end | Remove GBR QoS flow |
| Data cap reached | Reduce session AMBR |
| Handover | Update N3 tunnel endpoint |
| Edge breakout | Add UL classifier rule |
| Gaming mode | Adjust QoS parameters |

**Debugging modification:**

```bash
# Wireshark filter for modification messages
nas-5gs.sm.message_type == 0xc5 || nas-5gs.sm.message_type == 0xca || nas-5gs.sm.message_type == 0xcb

# N4 session modification
pfcp.message_type == 3 || pfcp.message_type == 4
```

**Key message types:**

| Message | Direction | Purpose |
|---------|-----------|---------|
| PDU Session Modification Request | UE→Network | UE wants change |
| PDU Session Modification Command | Network→UE | Network changes session |
| PDU Session Modification Complete | UE→Network | UE confirms command |
| PDU Session Modification Reject | Network→UE | Change denied |

</section>

<section class="lesson-pitfalls">

## Pitfalls

1. **Ignoring modification command.** When network sends modification command, UE must apply it. Ignoring can cause session release.

2. **QoS flow without packet filter.** Adding a QoS flow requires packet filter to classify traffic. Missing filter means traffic uses default flow.

3. **GBR request without resources.** Requesting guaranteed bitrate when network is congested → modification rejected.

4. **N4 update failure.** If SMF can't update UPF, modification fails. Check N4 connectivity.

5. **Forgetting RAN update.** Some modifications require gNB reconfiguration (new DRB). AMF must send N2 message.

6. **Policy timing issues.** PCF policy change may not reach SMF instantly. Stale policy can conflict with modification.

7. **Multi-access inconsistency.** UE on both 3GPP and non-3GPP must have consistent QoS rules across both accesses.

8. **Session AMBR vs UE AMBR.** Don't confuse session AMBR (per PDU session) with UE AMBR (across all sessions).
