## Why this matters

Traditional RAN deployments bind operators to single vendors for 7-10 years. This lock-in drives up costs, slows innovation, and limits deployment flexibility.

Understanding vendor lock-in is the first step toward evaluating Open RAN alternatives.

</section>

### Lock-in mechanisms

```
How Vendor Lock-in Happens:

┌─────────────────────────────────────────────────────────┐
│                                                          │
│  1. Proprietary Interfaces                              │
│  ┌─────────────────────────────────────────────────┐    │
│  │ RRU ←──── Custom CPRI ────→ BBU                 │    │
│  │                                                  │    │
│  │ Vendor A's CPRI extension ≠ Vendor B's          │    │
│  │ Cannot mix RRU from A with BBU from B           │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  2. Closed-Source Software                              │
│  ┌─────────────────────────────────────────────────┐    │
│  │ L1 PHY: Binary blob, no source                  │    │
│  │ L2 MAC: Proprietary scheduler                   │    │
│  │ L3 RRC: Vendor-specific extensions              │    │
│  │                                                  │    │
│  │ Cannot port to different hardware               │    │
│  │ Cannot modify behavior                          │    │
│  │ Cannot fix bugs yourself                        │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  3. Hardware Dependencies                               │
│  ┌─────────────────────────────────────────────────┐    │
│  │ Vendor A BBU = Custom ASIC/FPGA                 │    │
│  │ Software compiled for that specific HW          │    │
│  │                                                  │    │
│  │ Upgrade BBU = Buy new hardware                  │    │
│  │ Scale capacity = Buy vendor's expansion cards   │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  4. Data Model Lock-in                                  │
│  ┌─────────────────────────────────────────────────┐    │
│  │ Vendor A config format ≠ Vendor B config format │    │
│  │                                                  │    │
│  │ PM counters: proprietary naming                 │    │
│  │ Alarms: vendor-specific codes                   │    │
│  │ KPIs: non-standard definitions                  │    │
│  │                                                  │    │
│  │ OSS integration = custom adapters per vendor    │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

## Apply

**Lock-in assessment checklist:**

| Question | Red Flag |
|----------|----------|
| Can you mix RRU from vendor A with BBU from vendor B? | No |
| Are config/PM formats documented publicly? | No |
| Can you run software on commodity hardware? | No |
| Does contract allow 3rd-party integration? | Restricted |
| Is feature activation separate from HW purchase? | Bundled |
| Can you port investment to next-gen platform? | No |

**Negotiation leverage points:**

```bash
# Questions to ask vendor during RFP:

1. Interface documentation
   "Provide complete CPRI/fronthaul interface spec"
   → If refused, indicates proprietary lock-in intent

2. Data portability
   "Export all PM/CM data in standard format"
   → If non-standard, will cost to migrate

3. Software flexibility
   "Run your stack on our choice of COTS hardware"
   → If refused, hardware lock-in

4. Multi-vendor clause
   "Integrate with competitor's EMS via open API"
   → Resistance indicates closed ecosystem

5. Exit terms
   "What data/configs do we retain if we switch?"
   → Vague answer = trapped
```

</section>

<section class="lesson-pitfalls">

## Pitfalls

1. **"Open" claims.** Vendor claims "open" but only within their ecosystem. Demand proof of 3rd-party interop.

2. **Feature creep licensing.** Basic HW cheap, but every feature (CA, MIMO, beamforming) sold separately.

3. **Support dependency.** Cannot troubleshoot without vendor due to closed binaries. Operations team skill atrophies.

4. **Contract auto-renewal.** Multi-year contract renews automatically. Miss window, locked for another term.

5. **Training lock-in.** Staff trained only on one vendor's system. Institutional knowledge becomes vendor-specific.

6. **Integration debt.** Years of custom OSS adapters for one vendor. Switching means rewriting everything.

7. **Sunk cost fallacy.** "We've invested too much to switch." Continues paying premium indefinitely.

8. **Roadmap alignment.** Vendor's 5G timeline doesn't match operator's launch date. No alternatives.
