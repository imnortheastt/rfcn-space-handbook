# Runge-Kutta Methods

<section class="lesson-hook">

## Why Runge-Kutta?

Every orbit propagator, from student projects to NASA mission planning, relies on numerical integration. You cannot solve the three-body problem analytically. Atmospheric drag, solar radiation pressure, and gravitational harmonics make closed-form solutions impossible.

Runge-Kutta methods are the Swiss Army knife of ODE integration:
- **Simple to implement** — no matrix inversions, no implicit solves
- **Predictable accuracy** — clear error order relationships
- **Universally applicable** — works for any first-order ODE system

When you run STK, GMAT, or even a Kerbal Space Program mod, RK4 or its cousins are doing the heavy lifting.

</section>

<section class="lesson-intuition">

## The Core Idea: Weighted Slope Averaging

Euler's method uses one slope sample at the start of the interval:

```
         y
         |
         |    ● actual curve
         |   /
         | ●───────→ Euler step (overshoots)
         |/
         ●
         └────────────────── t
```

RK methods sample slopes at multiple points within the interval, then compute a weighted average:

```
         y
         |
         |        ● actual curve
         |      ●─┐
         |     /  │ k2: slope at midpoint
         |   ●────┴──→ RK2 step (closer!)
         |  /  k1
         ●
         └────────────────── t
```

More samples = better approximation = higher-order accuracy.

**Key insight**: We're not storing more history — we're evaluating the derivative function `f(t, y)` multiple times per step to capture curvature.

</section>

<section class="lesson-mechanism">

## The RK Family: From Euler to RK4

### RK1: Forward Euler (Order 1)

The simplest integrator. One evaluation per step.

$$y_{n+1} = y_n + h \cdot f(t_n, y_n)$$

**Error**: O(h) per step, O(1) globally — accumulates fast.

### RK2: Heun's Method (Order 2)

Two evaluations. Predict with Euler, correct with average slope.

$$k_1 = f(t_n, y_n)$$
$$k_2 = f(t_n + h, y_n + h \cdot k_1)$$
$$y_{n+1} = y_n + \frac{h}{2}(k_1 + k_2)$$

**Error**: O(h^2) per step — 100x smaller step = 10000x smaller error.

### RK2: Midpoint Method (Order 2)

Alternative RK2. Evaluate slope at interval midpoint.

$$k_1 = f(t_n, y_n)$$
$$k_2 = f(t_n + \frac{h}{2}, y_n + \frac{h}{2} k_1)$$
$$y_{n+1} = y_n + h \cdot k_2$$

### RK4: The Classic (Order 4)

The workhorse. Four evaluations per step.

$$k_1 = f(t_n, y_n)$$
$$k_2 = f(t_n + \frac{h}{2}, y_n + \frac{h}{2} k_1)$$
$$k_3 = f(t_n + \frac{h}{2}, y_n + \frac{h}{2} k_2)$$
$$k_4 = f(t_n + h, y_n + h \cdot k_3)$$
$$y_{n+1} = y_n + \frac{h}{6}(k_1 + 2k_2 + 2k_3 + k_4)$$

**Error**: O(h^4) per step. Halving step size = 16x accuracy improvement.

## Butcher Tableaux: The Compact Notation

Every RK method can be expressed as a Butcher tableau:

```
  c | A
  -----
    | b^T
```

Where:
- `c` = nodes (time offsets within step)
- `A` = stage coefficients (how to compute intermediate k values)
- `b` = weights (how to combine k values)

**RK4 Tableau:**

```
  0   |
  1/2 | 1/2
  1/2 | 0   1/2
  1   | 0   0   1
  ----+------------
      | 1/6 1/3 1/3 1/6
```

Reading the tableau:
- Row 1: k1 uses no previous k's (starts at t_n)
- Row 2: k2 at t + h/2, uses 0.5*k1
- Row 3: k3 at t + h/2, uses 0.5*k2
- Row 4: k4 at t + h, uses 1.0*k3
- Bottom: final weighted sum

## Error Analysis

| Method | Order | Local Error | Global Error | Evaluations |
|--------|-------|-------------|--------------|-------------|
| RK1    | 1     | O(h^2)      | O(h)         | 1           |
| RK2    | 2     | O(h^3)      | O(h^2)       | 2           |
| RK4    | 4     | O(h^5)      | O(h^4)       | 4           |

**Cost-benefit**: RK4 gives 4th order accuracy for only 4x the cost of Euler. Excellent deal.

## Python Implementation: Two-Body Orbit Propagation

```python
import numpy as np
import matplotlib.pyplot as plt

# Constants
MU = 398600.4418  # Earth gravitational parameter [km^3/s^2]

def two_body_ode(t, state):
    """
    Two-body equation of motion.
    state = [x, y, z, vx, vy, vz]
    Returns: [vx, vy, vz, ax, ay, az]
    """
    r = state[:3]
    v = state[3:]
    r_mag = np.linalg.norm(r)
    a = -MU * r / r_mag**3
    return np.concatenate([v, a])

def rk4_step(f, t, y, h):
    """Single RK4 step."""
    k1 = f(t, y)
    k2 = f(t + h/2, y + h/2 * k1)
    k3 = f(t + h/2, y + h/2 * k2)
    k4 = f(t + h, y + h * k3)
    return y + h/6 * (k1 + 2*k2 + 2*k3 + k4)

def propagate_orbit(state0, t_span, dt):
    """
    Propagate orbit using RK4.
    
    Args:
        state0: Initial [x, y, z, vx, vy, vz] in km, km/s
        t_span: (t_start, t_end) in seconds
        dt: Time step in seconds
    
    Returns:
        times, states arrays
    """
    t_start, t_end = t_span
    times = [t_start]
    states = [state0.copy()]
    
    t = t_start
    state = state0.copy()
    
    while t < t_end:
        h = min(dt, t_end - t)
        state = rk4_step(two_body_ode, t, state, h)
        t += h
        times.append(t)
        states.append(state.copy())
    
    return np.array(times), np.array(states)

# Example: ISS-like orbit
# Initial conditions: 400 km altitude circular orbit
r0 = 6778.0  # km (Earth radius + 400 km)
v0 = np.sqrt(MU / r0)  # Circular velocity

state0 = np.array([r0, 0, 0, 0, v0, 0])
period = 2 * np.pi * np.sqrt(r0**3 / MU)  # Orbital period

# Propagate for one orbit
times, states = propagate_orbit(state0, (0, period), dt=10.0)

# Check energy conservation
def specific_energy(state):
    r = np.linalg.norm(state[:3])
    v = np.linalg.norm(state[3:])
    return v**2/2 - MU/r

E0 = specific_energy(states[0])
E_final = specific_energy(states[-1])
print(f"Energy drift: {abs(E_final - E0) / abs(E0) * 100:.6f}%")
```

**Output** (typical):
```
Energy drift: 0.000003%
```

RK4 with 10-second steps maintains excellent energy conservation over one orbit.

<section class="lesson-pitfalls">

## Common Pitfalls

### 1. Step Size Too Large

RK4 is not magic. For highly eccentric orbits near periapsis, velocity changes rapidly.

**Fix**: Use adaptive stepping (see DOPRI lesson) or reduce step size near periapsis.

### 2. Assuming Higher Order = Always Better

RK4 is not always the best choice:
- Very smooth problems: lower order is cheaper
- Long integrations: symplectic methods preserve energy better
- Stiff problems: need implicit methods

### 3. Forgetting Units

Mixing km and m, or seconds and days, causes silent disasters.

```python
# BAD: Mixed units
MU = 3.986e14  # m^3/s^2
r0 = 6778.0    # km  <-- Unit mismatch!

# GOOD: Consistent units
MU = 398600.4418  # km^3/s^2
r0 = 6778.0       # km
```

### 4. Ignoring Error Accumulation

Local error O(h^5) does not mean errors vanish. Over 10000 orbits, small errors compound.

**Rule of thumb**: For N steps, global error ~ N * local_error.
