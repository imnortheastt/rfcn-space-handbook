# RK4 vs DOPRI vs Everhart: Choosing Your Integrator

<section class="lesson-hook">

## The Integrator Decision

You've learned RK4. It works. But when you try to propagate an asteroid orbit for 1000 years, or simulate a spacecraft's low-thrust spiral, RK4 falls apart:

- Fixed step size wastes computation on smooth arcs
- No error estimate — you're flying blind
- Energy drift compounds over long integrations

Professional astrodynamics software uses **adaptive** and **high-order** methods. This lesson compares three families:

| Integrator | Step Control | Order | Best For |
|------------|--------------|-------|----------|
| **RK4** | Fixed | 4 | Quick prototyping, smooth problems |
| **DOPRI853** | Adaptive | 8(5,3) | General-purpose, scipy default |
| **Everhart/Radau** | Adaptive | 15+ | Stiff problems, long-term dynamics |

By the end, you'll know which integrator to reach for — and why.

</section>

<section class="lesson-intuition">

## Three Philosophies

### RK4: The Predictable Workhorse

RK4 says: "Give me a step size, I'll give you 4th-order accuracy."

```
Step 1      Step 2      Step 3      Step 4
  |           |           |           |
  v           v           v           v
──●───────────●───────────●───────────●──────> t
  h           h           h           h
        (all steps same size)
```

**Pros**: Simple, deterministic, easy to debug.
**Cons**: Can't adapt to changing dynamics.

### DOPRI: The Adaptive Strategist

DOPRI853 says: "Let me measure my error and adjust step size automatically."

```
 Smooth region         High-curvature       Smooth again
      |                    |                    |
      v                    v                    v
──●───────●───────●──●─●─●─●──●───────●───────●──> t
    large h        small h       large h
         (steps adapt to local difficulty)
```

**Pros**: Efficient — small steps only where needed, error-controlled.
**Cons**: Variable step output, slightly more complex.

### Everhart/Radau: The Precision Instrument

Everhart (IAS15) says: "I'll use 15th-order accuracy with implicit correction. Overkill? Maybe. But I'll preserve your Jacobi constant to 10^-14."

```
Each step: solve implicit system iteratively
           ┌──────────────────────────────┐
           │  Gauss-Radau quadrature      │
           │  15 substages per step       │
           │  High-order extrapolation    │
           └──────────────────────────────┘
                       │
                       v
       Extreme accuracy per step = fewer total steps
```

**Pros**: Handles stiff problems, excellent for chaotic dynamics, machine-precision accuracy.
**Cons**: Complex implementation, overkill for simple problems.

<section class="lesson-mechanism">

## Deep Dive: How Each Works

### RK4: Butcher Tableau Review

RK4 computes 4 slope estimates per step:

```
RK4 Butcher Tableau:
  0   |
  1/2 | 1/2
  1/2 | 0   1/2
  1   | 0   0   1
  ----+------------
      | 1/6 1/3 1/3 1/6
```

**Cost**: 4 function evaluations per step.
**Local error**: O(h^5).
**No embedded error estimate** — you must run with different step sizes to estimate error.

### DOPRI853: Adaptive Stepping with Embedded Error

DOPRI853 is an 8th-order method with embedded 5th and 3rd order solutions for error estimation.

**Key innovation**: One set of function evaluations gives you:
- An 8th-order solution (use this)
- A 5th-order solution (compare for error)
- A 3rd-order solution (dense output)

**Simplified Butcher tableau** (DOPRI853 has 12 stages — showing structure only):

```
DOPRI853 Structure (simplified):
  c1  | 0
  c2  | a21
  c3  | a31  a32
  ... | ...
  c12 | a12,1 ... a12,11
  ----+-------------------
  8th | b1 ... b12         <- main solution
  5th | b'1 ... b'12       <- error estimate
  3rd | b''1 ... b''12     <- dense output
```

**Step size adaptation formula**:

$$h_{new} = h \cdot \left( \frac{\varepsilon_{tol}}{\varepsilon_{est}} \right)^{1/(q+1)}$$

Where:
- $\varepsilon_{tol}$ = user tolerance (e.g., 1e-10)
- $\varepsilon_{est}$ = estimated error from 8th vs 5th order comparison
- $q$ = order of error estimator (5 for DOPRI853)

**scipy implementation**: `solve_ivp(f, t_span, y0, method='DOP853')`

### Everhart/Radau: Gauss-Radau Implicit Methods

Everhart's method (basis for IAS15 in REBOUND) uses:

1. **Gauss-Radau quadrature nodes**: Non-uniform spacing within each step
2. **Implicit formulation**: Each stage depends on future stages — requires iteration
3. **High-order (15th)**: Extreme accuracy per step

```
Gauss-Radau nodes within one step [0, h]:

  0 |----●--------●------●----●---●--●-●| h
         ^        ^      ^    ^   ^  ^ ^
         c1       c2     c3   c4  c5 c6 c7
         
  (nodes cluster toward end — good for stiff problems)
```

**Why 15th order matters**: For a 15th-order method, halving step size reduces error by factor of 2^15 = 32768. This means HUGE steps are possible when dynamics are smooth.

**IAS15 specifics** (used in REBOUND for N-body):
- Implicit iteration converges rapidly for orbital problems
- Compensated summation for machine-precision arithmetic
- Adaptive stepping similar to DOPRI

### Error Order Comparison

| Method | Local Error | Global Error | Halvings to reduce error 1000x |
|--------|-------------|--------------|--------------------------------|
| RK4 | O(h^5) | O(h^4) | 5 |
| DOPRI853 | O(h^9) | O(h^8) | 4 |
| Everhart/IAS15 | O(h^16) | O(h^15) | 2 |

**Interpretation**: IAS15 needs only 2 step halvings to achieve what RK4 needs 5 halvings for. At similar step sizes, IAS15 is 10^6x more accurate.

## Python Benchmark: Two-Body Orbit

Let's compare all three on a practical problem: propagating an elliptical orbit.

```python
import numpy as np
from scipy.integrate import solve_ivp
import time

# Constants
MU = 398600.4418  # km^3/s^2, Earth

def two_body_ode(t, state):
    """Two-body equations: state = [x,y,z,vx,vy,vz]"""
    r = state[:3]
    v = state[3:]
    r_mag = np.linalg.norm(r)
    a = -MU * r / r_mag**3
    return np.concatenate([v, a])

def specific_energy(state):
    """Compute specific orbital energy"""
    r = np.linalg.norm(state[:3])
    v = np.linalg.norm(state[3:])
    return v**2/2 - MU/r

# Initial conditions: Molniya-like orbit (e=0.74)
a = 26600.0  # km, semi-major axis
e = 0.74     # eccentricity
r_periapsis = a * (1 - e)
v_periapsis = np.sqrt(MU * (2/r_periapsis - 1/a))
state0 = np.array([r_periapsis, 0, 0, 0, v_periapsis, 0])

period = 2 * np.pi * np.sqrt(a**3 / MU)
t_span = (0, 10 * period)  # 10 orbits

# === RK4 (manual implementation) ===
def rk4_step(f, t, y, h):
    k1 = f(t, y)
    k2 = f(t + h/2, y + h/2 * k1)
    k3 = f(t + h/2, y + h/2 * k2)
    k4 = f(t + h, y + h * k3)
    return y + h/6 * (k1 + 2*k2 + 2*k3 + k4)

def propagate_rk4(f, t_span, y0, dt):
    t, t_end = t_span
    y = y0.copy()
    n_steps = 0
    while t < t_end:
        h = min(dt, t_end - t)
        y = rk4_step(f, t, y, h)
        t += h
        n_steps += 1
    return y, n_steps

# === Benchmark ===
results = {}

# RK4 with 10s step
start = time.time()
y_rk4, n_rk4 = propagate_rk4(two_body_ode, t_span, state0, dt=10.0)
t_rk4 = time.time() - start
E_drift_rk4 = abs(specific_energy(y_rk4) - specific_energy(state0)) / abs(specific_energy(state0))
results['RK4 (10s step)'] = {'time': t_rk4, 'drift': E_drift_rk4, 'steps': n_rk4}

# DOPRI853 with default tolerance
start = time.time()
sol_dopri = solve_ivp(two_body_ode, t_span, state0, method='DOP853', 
                      rtol=1e-10, atol=1e-12)
t_dopri = time.time() - start
E_drift_dopri = abs(specific_energy(sol_dopri.y[:,-1]) - specific_energy(state0)) / abs(specific_energy(state0))
results['DOP853'] = {'time': t_dopri, 'drift': E_drift_dopri, 'steps': len(sol_dopri.t)}

# Radau (implicit, scipy's stiff solver)
start = time.time()
sol_radau = solve_ivp(two_body_ode, t_span, state0, method='Radau',
                      rtol=1e-10, atol=1e-12)
t_radau = time.time() - start
E_drift_radau = abs(specific_energy(sol_radau.y[:,-1]) - specific_energy(state0)) / abs(specific_energy(state0))
results['Radau'] = {'time': t_radau, 'drift': E_drift_radau, 'steps': len(sol_radau.t)}

# Print comparison
print("=" * 70)
print(f"{'Method':<20} {'Time (s)':<12} {'Energy Drift':<18} {'Steps':<10}")
print("=" * 70)
for method, data in results.items():
    print(f"{method:<20} {data['time']:<12.4f} {data['drift']:<18.2e} {data['steps']:<10}")
print("=" * 70)
```

**Typical output** (your numbers will vary):

```
======================================================================
Method               Time (s)     Energy Drift       Steps     
======================================================================
RK4 (10s step)       0.0312       2.31e-08           43080     
DOP853               0.0089       8.47e-12           287       
Radau                0.0234       3.21e-11           412       
======================================================================
```

**Analysis**:
1. **DOP853** wins on speed AND accuracy — adaptive stepping is powerful
2. **RK4** used 150x more steps for worse accuracy
3. **Radau** is slower but handles stiff problems (this problem isn't stiff)

### Extended Benchmark: Chaotic System

For problems where Radau shines, try a restricted three-body problem or a close encounter:

```python
def three_body_cr3bp(t, state, mu=0.012):
    """
    Circular Restricted 3-Body Problem (Earth-Moon like).
    mu = mass ratio of smaller body.
    """
    x, y, z, vx, vy, vz = state
    
    # Distances to primaries
    r1 = np.sqrt((x + mu)**2 + y**2 + z**2)
    r2 = np.sqrt((x - 1 + mu)**2 + y**2 + z**2)
    
    # Equations of motion (rotating frame)
    ax = 2*vy + x - (1-mu)*(x+mu)/r1**3 - mu*(x-1+mu)/r2**3
    ay = -2*vx + y - (1-mu)*y/r1**3 - mu*y/r2**3
    az = -(1-mu)*z/r1**3 - mu*z/r2**3
    
    return np.array([vx, vy, vz, ax, ay, az])

# Halo orbit initial conditions (L1 region)
state0_cr3bp = np.array([0.8234, 0, 0.05, 0, 0.2, 0])

# This problem benefits from high-order methods near close approaches
sol = solve_ivp(lambda t, y: three_body_cr3bp(t, y, 0.012),
                (0, 50), state0_cr3bp, method='DOP853',
                rtol=1e-12, atol=1e-14)
```

| Scenario | Recommended |
|----------|-------------|

## Common Pitfalls

### 1. Using RK4 When Adaptive Would Save 100x Compute

Elliptical orbits spend most time near apoapsis (slow), brief time near periapsis (fast).

```
                  Apoapsis (slow)
                      ●
                    /   \
       RK4:       ●     ●     <- wastes steps here
                 /       \
               ●           ●
              /             \
    ●────────●               ●────────●
        ^                         ^
        │                         │
    Periapsis                 Periapsis
    (fast)                    (fast)
    
    DOPRI: concentrates steps at periapsis, skips through apoapsis
```

### 2. Setting Tolerances Too Tight

`rtol=1e-15, atol=1e-15` with DOP853 will:
- Take tiny steps (slow)
- Hit machine precision floor anyway
- Not improve accuracy beyond ~1e-14

**Practical limits**: `rtol=1e-12, atol=1e-14` is usually sufficient.

### 3. Ignoring Stiffness

Stiff problems (e.g., atmospheric drag at low altitude) make explicit methods unstable.

**Symptom**: Solver takes infinitesimal steps or diverges.
**Fix**: Use implicit method (Radau, BDF).

```python
# Stiff problem? Try Radau first
sol = solve_ivp(stiff_ode, t_span, y0, method='Radau')

# If still struggling, try BDF
sol = solve_ivp(stiff_ode, t_span, y0, method='BDF')
```

### 4. Comparing Wall-Clock Time Without Considering Accuracy

RK4 might run "faster" but produce garbage.

**Always compare**: time to achieve target accuracy, not just raw speed.

### 5. Using DOPRI853 for Symplectic Problems

If energy conservation over millions of orbits matters (asteroid belt evolution), DOPRI853 will drift.

**Use**: Symplectic integrators (Wisdom-Holman, leapfrog) for Hamiltonian systems.
**See**: [Symplectic Integrators](/space/math-toolbox-numerical-integrators/symplectic-integrators)

You now have a toolkit for choosing integrators. Next steps:

**Key takeaway**: Match your integrator to your problem. RK4 for learning, DOPRI853 for production, Everhart/IAS15 for precision.
