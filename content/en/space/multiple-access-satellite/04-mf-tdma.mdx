## Why this matters

MF-TDMA is the workhorse of modern VSAT systems. It combines FDMA (multiple frequency carriers) with TDMA (time slots on each carrier). The result: a network management system can dynamically assign any terminal to any slot on any carrier, matching capacity to demand in real-time. This is why your Viasat or HughesNet connection can burst to high speeds when needed but share resources efficiently across millions of subscribers. DVB-RCS2, the global standard, is built on MF-TDMA.

### Dynamic bandwidth allocation

```
Why MF-TDMA Beats Pure TDMA or FDMA:

┌─────────────────────────────────────────────────────────────────────┐
│                                                                      │
│  SCENARIO: 100 terminals, mostly idle, occasional bursts           │
│                                                                      │
│  PURE FDMA:                                                         │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  Each terminal gets fixed 360 kHz (36 MHz / 100)            │   │
│  │  Peak rate: 360 kHz × 2 bps/Hz = 720 kbps                   │   │
│  │  If 5% active: 95% of bandwidth wasted                      │   │
│  │  NO BURSTING - stuck at 720 kbps                            │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  PURE TDMA (single carrier):                                        │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  36 MHz carrier, 100 slots per frame                        │   │
│  │  Each terminal: 1 slot = 1% of time at 72 Mbps              │   │
│  │  Peak rate: 720 kbps sustained                              │   │
│  │  Inefficient for small terminals (big preamble overhead)    │   │
│  │  LIMITED BURSTING - constrained by slot size                │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  MF-TDMA:                                                           │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  4 carriers × 6 MHz, 25 slots per carrier per frame         │   │
│  │  Small terminal: 1 slot on 1 carrier = 360 kbps             │   │
│  │  Medium burst: 4 slots across 2 carriers = 2.9 Mbps         │   │
│  │  Large burst: 20 slots across 4 carriers = 14 Mbps          │   │
│  │  DYNAMIC - grab as many slots as needed!                    │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  MF-TDMA provides flexible bursting while efficiently sharing      │
│  bandwidth across terminals with different needs.                  │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### Bandwidth allocation mechanisms

```
MF-TDMA Capacity Assignment:

┌─────────────────────────────────────────────────────────────────────┐
│                                                                      │
│  THREE TYPES OF CAPACITY:                                          │
│                                                                      │
│  1. CRA (Constant Rate Assignment)                                 │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • Fixed slots every frame (like SCPC, but shared)            │ │
│  │  • Guaranteed bandwidth, highest priority                     │ │
│  │  • Used for: VoIP, video conferencing, critical data         │ │
│  │  • Example: 256 kbps CRA for enterprise voice                │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  2. RBDC (Rate-Based Dynamic Capacity)                             │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • Terminal requests rate (kbps), hub assigns slots           │ │
│  │  • Persists until terminal requests lower rate                │ │
│  │  • Used for: Streaming, downloads, interactive data          │ │
│  │  • Example: Request 2 Mbps for video streaming               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  3. VBDC (Volume-Based Dynamic Capacity)                           │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • Terminal requests volume (bytes), hub assigns slots        │ │
│  │  • One-shot: capacity granted once per request                │ │
│  │  • Used for: Email, web browsing, bursts                     │ │
│  │  • Example: Request 50 KB for HTTP response                  │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  4. FCA (Free Capacity Assignment)                                 │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • Hub distributes unused capacity to active terminals        │ │
│  │  • Best-effort, lowest priority                               │ │
│  │  • Absorbs CRA + RBDC + VBDC leftovers                       │ │
│  │  • Used for: Background transfers, software updates          │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  TOTAL CAPACITY = CRA + RBDC + VBDC + FCA ≤ System capacity       │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### Terminal burst time plan (TBTP)

```
How Slots Are Assigned:

┌─────────────────────────────────────────────────────────────────────┐
│                                                                      │
│  FORWARD LINK (Hub → Terminals):                                   │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                                │ │
│  │  Hub broadcasts TBTP (Terminal Burst Time Plan):              │ │
│  │                                                                │ │
│  │  "In superframe 1234:                                         │ │
│  │     Terminal 001: Carrier 2, Slots 5-7                       │ │
│  │     Terminal 002: Carrier 1, Slot 12                         │ │
│  │     Terminal 003: Carrier 3, Slots 1-4, 8-10                 │ │
│  │     ..."                                                      │ │
│  │                                                                │ │
│  │  Sent 2-3 superframes in advance (scheduling latency)        │ │
│  │                                                                │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  RETURN LINK (Terminals → Hub):                                    │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                                │ │
│  │  Terminals transmit capacity requests via:                   │ │
│  │  • SAC (Satellite Access Control) field in each burst        │ │
│  │  • Dedicated contention slots (for initial access)           │ │
│  │                                                                │ │
│  │  Request format:                                              │ │
│  │    "Terminal 001 requests: CRA=128kbps, RBDC=0, VBDC=8KB"   │ │
│  │                                                                │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  TIMING LOOP:                                                       │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                                │ │
│  │  Superframe N:   Terminal sends capacity request              │ │
│  │  Superframe N+1: Hub processes, computes TBTP                │ │
│  │  Superframe N+2: Hub broadcasts TBTP                         │ │
│  │  Superframe N+3: Terminal transmits in assigned slots        │ │
│  │                                                                │ │
│  │  Total delay: ~3 superframes = 150-500 ms typical            │ │
│  │                                                                │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

import numpy as np

def mf_tdma_frame_capacity(num_carriers, carrier_bw_mhz, symbol_rate_msps,
                           bits_per_symbol, code_rate, slots_per_frame,
                           preamble_symbols, guard_symbols):
    """
    Calculate MF-TDMA frame capacity.
    """
    symbols_per_slot = (carrier_bw_mhz * 1e6 / symbol_rate_msps / 1e6 / 
                        slots_per_frame * symbol_rate_msps * 1e6 / 1e6)
    
    slot_duration_us = 1000000 / symbol_rate_msps / slots_per_frame
    
    payload_symbols = int(slot_duration_us * symbol_rate_msps) - preamble_symbols - guard_symbols
    
    bits_per_slot = payload_symbols * bits_per_symbol * code_rate
    bytes_per_slot = bits_per_slot / 8
    
    total_slots = num_carriers * slots_per_frame
    
    frame_duration_ms = 1000 / symbol_rate_msps * slots_per_frame * (preamble_symbols + guard_symbols + payload_symbols) / 1000
    
    return {
        "num_carriers": num_carriers,
        "slots_per_frame": slots_per_frame,
        "total_slots": total_slots,
        "bytes_per_slot": int(bytes_per_slot),
        "payload_symbols": payload_symbols
    }

def capacity_allocation(total_slots, terminals):
    """
    Simulate MF-TDMA capacity allocation.
    
    Args:
        total_slots: Total slots per superframe
        terminals: List of dicts with {id, cra_slots, rbdc_slots, vbdc_slots}
    
    Returns:
        Allocation results
    """
    cra_total = sum(t.get("cra_slots", 0) for t in terminals)
    rbdc_total = sum(t.get("rbdc_slots", 0) for t in terminals)
    vbdc_total = sum(t.get("vbdc_slots", 0) for t in terminals)
    
    committed = cra_total + rbdc_total + vbdc_total
    fca_available = max(0, total_slots - committed)
    
    active_terminals = len([t for t in terminals if 
                           t.get("cra_slots", 0) + t.get("rbdc_slots", 0) + t.get("vbdc_slots", 0) > 0])
    
    fca_per_terminal = fca_available // max(1, active_terminals) if active_terminals > 0 else 0
    
    return {
        "total_slots": total_slots,
        "cra_slots": cra_total,
        "rbdc_slots": rbdc_total,
        "vbdc_slots": vbdc_total,
        "fca_slots": fca_available,
        "utilization": committed / total_slots,
        "fca_per_terminal": fca_per_terminal
    }

def terminal_throughput(cra_kbps, rbdc_kbps, vbdc_burst_kb, fca_kbps,
                        cir_kbps, mir_kbps):
    """
    Calculate terminal throughput with QoS constraints.
    
    Args:
        cra_kbps: Constant rate allocation
        rbdc_kbps: Rate-based dynamic capacity
        vbdc_burst_kb: Volume-based burst (one-shot)
        fca_kbps: Free capacity estimate
        cir_kbps: Committed Information Rate
        mir_kbps: Maximum Information Rate
    
    Returns:
        Effective throughput
    """
    total = cra_kbps + rbdc_kbps + fca_kbps
    
    guaranteed = min(cra_kbps, cir_kbps)
    
    peak = min(total, mir_kbps)
    
    return {
        "guaranteed_kbps": guaranteed,
        "typical_kbps": cra_kbps + rbdc_kbps,
        "peak_kbps": peak,
        "vbdc_burst_kb": vbdc_burst_kb
    }

def compare_access_methods(num_terminals, activity_factor, 
                           peak_rate_kbps, system_capacity_mbps):
    """
    Compare FDMA, TDMA, and MF-TDMA for a given scenario.
    """
    fdma_per_terminal = system_capacity_mbps * 1000 / num_terminals
    fdma_effective = fdma_per_terminal * activity_factor
    fdma_peak = fdma_per_terminal  # No bursting
    
    tdma_frame_eff = 0.85
    tdma_capacity = system_capacity_mbps * 1000 * tdma_frame_eff
    tdma_per_terminal = tdma_capacity / num_terminals
    tdma_peak = tdma_capacity / (num_terminals * activity_factor)  # Burst to full
    
    mf_tdma_eff = 0.88
    mf_tdma_capacity = system_capacity_mbps * 1000 * mf_tdma_eff
    mf_tdma_burst_ratio = 1 / activity_factor
    mf_tdma_peak = min(mf_tdma_capacity * 0.5, peak_rate_kbps * mf_tdma_burst_ratio)
    
    return {
        "fdma": {"sustained": fdma_per_terminal, "peak": fdma_peak},
        "tdma": {"sustained": tdma_per_terminal, "peak": min(tdma_peak, tdma_capacity * 0.3)},
        "mf_tdma": {"sustained": mf_tdma_capacity / num_terminals, "peak": mf_tdma_peak}
    }

<section class="lesson-pitfalls">

## Pitfalls

1. **Ignoring scheduling latency.** MF-TDMA adds 2-3 superframes of delay for capacity requests. First-packet latency can be 100-200 ms higher than SCPC.

2. **Over-subscribing CRA.** CRA is guaranteed — if you promise more CRA than system capacity, you'll fail at peak load. Keep total CRA < 70% of capacity.

3. **Expecting instant bursts.** VBDC/RBDC requests take 2-3 superframes to fulfill. Bursty web traffic feels this latency.

4. **Forgetting rain fade margin.** MF-TDMA systems use adaptive coding/modulation. Heavy rain reduces slots per terminal — budget for degraded mode.

5. **Underestimating preamble overhead.** Small slots have higher preamble percentage. 64-byte slots with 96-symbol preambles lose ~20% to overhead.

</section>
