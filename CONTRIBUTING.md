# Contributing to RF · Core Network · Space Handbook

Thank you for contributing. This guide covers the author workflow for lesson writers
and platform contributors alike.

---

## Branch Model

- `main` — production. Every commit on `main` is or will be deployed.
- Feature/content branches — branch off `main`, open a PR when ready.
- Branch naming: `feat/<slug>`, `fix/<slug>`, `content/<lesson-id>`.
- Never push directly to `main`. All changes land via PR with at least one review.

---

## Lesson Parity Rule

**Lessons land in pairs or not at all.**

Every lesson must exist in both Vietnamese (`lang: vi`) and English (`lang: en`)
before it is merged. Both files share the same `parityId` in their frontmatter.
A PR containing only one language of a lesson pair will not be merged.

This is non-negotiable — it is the core promise of the platform to bilingual readers.

---

## Commit Conventions

We use [Conventional Commits](https://www.conventionalcommits.org/):

```
feat(rf):     add lesson on antenna radiation patterns
fix(widget):  correct link budget formula in FresnelZone widget
content(core): add vi+en pair for 5G SBA lesson unit-03
docs:         update blueprint §6 sourcing policy
refactor(ui): extract CitationBlock into shared component
```

- Scope is optional but encouraged (domain: `rf`, `core`, `space`, `widget`, `ui`).
- No AI tool references in commit messages (e.g., do not write "generated by Claude").
- Keep commits focused — one logical change per commit.

---

## Adding a Lesson

1. Copy the lesson skeleton from `docs/PLATFORM_BLUEPRINT.md` Appendix A.
2. Write both `vi` and `en` versions with identical `parityId`.
3. Place files under `content/<domain>/<track>/`.
4. Add any new citations to `data/citations.json`.
5. Add new glossary terms to `data/glossary.json`.

---

## Adding a Citation or Glossary Term

- Citations: append to `data/citations.json` following the existing schema.
- Glossary: append to `data/glossary.json` with `term`, `definition`, `lang`, and `seeAlso`.
- Use 3GPP TS/TR with release numbers; IEEE with year; IETF RFC with number.

## Running Validators Locally

```bash
pnpm install
pnpm lint          # ESLint + Prettier check
pnpm typecheck     # TypeScript
pnpm test          # Unit tests
pnpm build         # Full production build (catches Astro content errors)
```

All checks must pass before opening a PR.
